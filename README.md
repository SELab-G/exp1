# گزارش پیاده‌سازی آزمایش ۱ - درس آزمایشگاه مهندسی نرم‌افزار

این پروژه به منظور تمرین مفاهیم پایه Git، GitFlow و استقرار سایت با GitHub Pages در قالب آزمایش ۱ درس آزمایشگاه مهندسی نرم‌افزار ایجاد شده است.

## برنچ‌های استفاده‌شده

| برنچ | هدف برنچ | وضعیت مرج |
|------|----------|-----------|
| main | برنچ اصلی تولید (Production) - نسخه پایدار و نهایی پروژه (محافظت شده با PR) | فعال |
| develop | برنچ توسعه - تمام ویژگی‌ها ابتدا در این شاخه ادغام می‌شوند | فعال |
| feature/contact-page | افزودن صفحه تماس (contact.html) و ناوبری اولیه توسط نفر اول | مرج شده به develop سپس به main |
| feature/about-page | افزودن صفحه درباره ما (about.html) توسط نفر دوم (ایجاد Conflict اول) | مرج شده به develop سپس به main |
| feature/french-language | افزودن پشتیبانی زبان فرانسوی (دکمه تغییر زبان و ترجمه) توسط نفر دوم (ایجاد Conflict دوم) | مرج شده به develop سپس به main |

تمامی برنچ‌های feature پس از تکمیل به develop مرج شده‌اند و سپس با یک Pull Request نهایی به main منتقل شده‌اند. استقرار خودکار با استفاده از GitHub Actions انجام می‌شود.

## رفع Merge Conflict‌ها در طول پروژه

در طول توسعه پروژه، دو بار با Merge Conflict واقعی مواجه شدیم و آن‌ها را به صورت دستی و اصولی رفع کردیم:

### ۱. Merge Conflict در فایل index.html (مرج برنچ feature/about-page به develop)

زمانی که نفر دوم سعی کرد برنچ feature/about-page را به develop merge کند، Conflict ایجاد شد. دلیل: نفر اول قبلاً در برنچ feature/contact-page ناوبری را تغییر داده و لینک Contact را اضافه کرده بود. نفر دوم نیز همان قسمت ناوبری را تغییر داده و لینک About را اضافه کرده بود.

فایل‌های درگیر: index.html (بخش nav)

راه‌حل: 
- هر دو تغییر لازم بودند (هم لینک Contact و هم لینک About).
- بخش‌های <<<<<<< HEAD، ======= و >>>>>>> را به صورت دستی ترکیب کردیم.
- ناوبری نهایی به این شکل درآمد:

<nav>
    <a href="index.html">Home</a>
    <a href="about.html">About</a>
    <a href="contact.html">Contact</a>
</nav>

کامیت رفع کانفلیکت: Merge branch 'feature/about-page' into develop (resolve conflicts in index.html)

### ۲. Merge Conflict در فایل‌های index.html و script.js (مرج برنچ feature/french-language به develop)

بعد از اینکه برنچ‌های قبلی با موفقیت merge شدند، نفر دوم برنچ feature/french-language را برای اضافه کردن قابلیت زبان فرانسه ساخت. هنگام merge این برنچ به develop، دو Conflict ایجاد شد:

فایل‌های درگیر: 
- index.html (اضافه شدن دکمه‌های تغییر زبان در کنار ناوبری موجود)
- script.js (اضافه شدن شیء translations و تابع updateLanguage در کنار کدهای قبلی)

راه‌حل: 
- در index.html: دکمه‌های EN/FR را بدون حذف ناوبری به بالای صفحه اضافه کردیم.
- در script.js: توابع جدید ترجمه را بدون حذف یا تداخل با کدهای قبلی (سخنرانان و برنامه) اضافه کردیم. کدهای قدیمی حفظ شدند و توابع جدید در کنار آنها قرار گرفتند.

کامیت رفع کانفلیکت: Merge branch 'feature/french-language' into develop (resolve conflicts in index.html and script.js)

با رفع این دو کانفلیکت، پروژه بدون از دست رفتن هیچ تغییری به‌روزرسانی شد و همه قابلیت‌ها (Contact Page، About Page، زبان فرانسه) با موفقیت در شاخه develop و سپس main ادغام شدند.

## ۱) پوشه‌ی .git چیست؟ چه اطلاعاتی در آن ذخیره می‌شود؟ با چه دستوری ساخته می‌شود؟

وقتی داخل یک پوشه دستور git init یا وقتی git clone می‌کنید، این پوشه ساخته می‌شود و همه‌ی داده‌های مربوط به تاریخچه، پیکربندی و وضعیت داخلی مخزن را نگه می‌دارد.

**چه چیزهایی داخل .git هست؟**

- objects/ — تمام اشیاء گیت: blobها (محتوای فایل‌ها)، treeها (درخت فایل/پوشه)، و commitها.
- refs/ — اشاره‌گرها: شاخه‌ها (refs/heads/)، تگ‌ها (refs/tags/) و ریموت‌ها (refs/remotes/).
- HEAD — نشان‌دهنده‌ی شاخه یا commit فعلی.
- index — فایل stage یا همان staging area (فایل باینری که وضعیت staged را نگهداری می‌کند).
- config — تنظیمات محلی مخزن (میشود اینجا تنظیمات repo-specific را دید).
- logs/ — تاریخچه‌ی حرکت refs (مثلا reflog برای بازگردانی).
- hooks/ — اسکریپت‌هایی که موقع رویدادهای خاص اجرا می‌شوند (قبل از commit، قبل از push و...).

**با چه دستوری ساخته می‌شود؟**

```
git init
```

یا وقتی مخزن را از راه دور کپی می‌کنید:

```
git clone <url>
```

## ۲) منظور از atomic بودن در atomic commit و atomic pull-request چیست؟

هنگامی که از واژه atomic برای یک عملیات استفاده می‌کنیم، یعنی یک عملیات یا کاملا اجرا می‌شود یا اصلا اجرا نمی‌شود. در **Atomic commit** هر commit در گیت یک واحد atomic تغییر است؛ یا کل تغییرات آن commit وارد تاریخچه می‌شود یا اصلا وارد نمی‌شود. در **Atomic pull-request** (یا merge از طریق PR) وقتی PR ادغام می‌شود، همه تغییرات آن PR به‌طور یکجا وارد شاخه‌ی هدف می‌شود یا اگر مشکلی (مثلا تست‌ها شکست بخورند) هست، هیچ‌کدام وارد نمی‌شوند. در سرویس‌هایی مثل GitHub/GitLab، عملیات merge معمولا به‌صورت atomic انجام می‌شود. هدف atomic بودن جلوگیری از حالات نیمه‌پایدار در کد شاخه‌ی اصلی و تسهیل بازگشت به وضعیت سالم است.
